# Support

`codex-app-web` is an independent, community-maintained project. Support is
best-effort and happens in public unless the report contains sensitive security
details or private information.

## Where To Ask

- Use GitHub Discussions for usage questions, setup help, and design questions.
- Use issues for reproducible bugs, feature requests, documentation gaps, and
  release blockers.
- Use private GitHub Security Advisories for vulnerabilities, exploit details,
  secrets, private logs, or reports that could help attackers.
- Do not paste tokens, private prompts, credentials, cookies, local transcripts,
  or sensitive screenshots into public issues, discussions, or pull requests.

## Maintainer Response Targets

These targets are not paid support guarantees. They are triage goals for normal
repository conditions:

| Priority      | Initial maintainer response target | Typical handling                                                     |
| ------------- | ---------------------------------- | -------------------------------------------------------------------- |
| `priority:P0` | 2 business days                    | Security, release, public install, CI, or package integrity blockers |
| `priority:P1` | 5 business days                    | Major compatibility, product, governance, or operational gaps        |
| `priority:P2` | 10 business days                   | Quality, test, DX, or maintainability work                           |
| `priority:P3` | 20 business days                   | Polish, demo, community, or future roadmap work                      |

If a report needs information from the reporter, maintainers apply
`needs-info` when that label exists or leave a clear comment. Issues with no
reporter response for 14 days after that comment may be closed as inactive.

## Stale Policy

Stale handling is manual unless a reviewed workflow is added later.

- P0, security, release blocker, and active regression issues are not closed as
  stale.
- Untriaged issues should receive area, type, priority, and risk labels before
  stale status is considered.
- Inactive P2/P3 issues may be marked stale after 60 days without activity.
- Stale issues may be closed after 14 more days if no new evidence or owner is
  added.
- Closed stale issues may be reopened when fresh reproduction steps, logs, or a
  concrete implementation path are available.
